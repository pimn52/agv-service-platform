## 着陆页

- **定位**：未登录用户的唯一入口。演示体验（推荐）和登录/注册两个按钮并排。
- **交互演变**：原设计为 3 秒自动跳转演示模式，数字跳动（钩子效果）仅需维持 3 秒注意力。后因加入真实注册用户支持改为手动入口，用户停留时间变长。
- **数据策略**：三组数字不使用虚假运营数据，改为行业公开真实数据——16,000+ 累计部署 L4 无人车（新石器 CES 2026）、1 亿+ 自动驾驶里程 km（新石器 2026.2 公告）、60+ 覆盖城市（九识智能 2025）。数据来源可查证，避免"假数据经不起审视"的信任问题。
- **组件**：`src/components/landing/landing-page.tsx`，`AnimatedNumber` 做缓动数字动画

## 设计原则

**业务线感知是横向的，页面功能是纵向的。** 三大业务线贯穿所有一级页面，但展示深度随页面职能变化：

| 页面 | 职能 | 展示逻辑 |
|------|------|----------|
| **首页** | 实时操作 + 下单入口 | 在途订单动态面板（Tab × 业务线）+ 两张下单卡片常驻 |
| **服务跟踪** | 实时追踪 | 下拉列表切换在途订单，按服务类型渲染专属 Dashboard |
| **订单列表** | 历史管理 | 全部/进行中/已完成筛选，列表项按服务类型区分图标与摘要 |
| **我的** | 账户 + 运营数据 | 运营看板（Tab × 业务线）+ 功能菜单 |

## 首页

- **下单卡片**：两块入口（物流配送 / 巡游租车），紧凑但不隐藏，始终可见。不使用的业务不下架——它存在本身就是"平台能力"的无声告知
- **在途订单动态面板**：Tab 按物流/贩卖/安防分组，每个 Tab 下展示该业务线的活跃订单卡片 + 操作按钮 + 通知
- **Tab 默认定位**：首次加载定位到有 actionable 消息的 Tab（当前逻辑）；后续可演进为仅显示用户有订单的 Tab

## 服务跟踪页

- 顶部下拉列表列出所有在途订单（跨业务线），选中后渲染对应业务的追踪 Dashboard
- 物流：路线地图 + 经停点时间线 + 剩余距离/时间
- 贩卖：实时销售数据 + 库存图表 + 销售趋势
- 安防：巡检进度 + 设备状态 + 告警列表
- 单用户通常只有 1-2 个在途订单，下拉列表不会过长

## 订单列表页

- 全部/进行中/已完成三态筛选
- 列表项按服务类型展示：物流显示起止地址 + 运单数、贩卖/安防显示取车地点 + 时长
- 进行中订单点击跳转服务跟踪，已完成订单点击查看详情

## 我的页面

- **运营看板**（新增）：用户卡片与功能菜单之间，2×2 Grid 四指标卡片
  - Tab 按业务线分面：物流/贩卖/安防各自四个指标
  - 物流：本月运费 + 准时送达率 + 异常率 + **在途运单**（实时）
  - 贩卖：本月销售额 + 单车日均销 + 库存周转 + **在售车辆**（实时）
  - 安防：巡检覆盖率 + 设备在线率 + 异常告警 + **在巡车辆**（实时）
  - "在途/在售/在巡"指标从 store 实时取，演示中操作订单后切回"我的"可看到数字变化
- **Tab 智能显示**：
  - 新用户（0 订单）：三个 Tab 全显示，未开通业务 Tab 内展示服务介绍卡片（宣传引导）
  - 单一业务用户（如仅物流）：默认展示物流 Tab；贩卖/安防折叠为一行小字"探索更多服务 ›"
  - 多业务用户：有订单的 Tab 正常展示，无订单的 Tab 折叠
- **功能菜单**：我的订单 / 地址管理 / 发票管理 / 设置等（保持不变）

## L2 子 Tab 架构（多车/多运单切换）

> 整车（FTL）和散件（LTL）在追踪页和首页均使用 L2 胶囊 Tab 切换车辆/运单。对齐后规则统一。

| 维度 | FTL | LTL |
|------|-----|-----|
| **L2 Tab 触发** | ftlWaybills.length > 1 | `ensureLTLCompartments(order)` 去重 vehicleId > 1 |
| **Tab 标签** | `运单N · 车牌号` | `车牌 · N张运单`（多运单）/ `车牌`（单运单） |
| **Tab 内过滤** | 选中 → 时间线/交接记录/操作均按该运单过滤 | 选中 → 时间线/交接记录/操作均按该车牌过滤 |
| **"多运单"判定** | N/A | 订单级：`order.ltlWaybills.length > 1`，不分子 Tab 内运单数 |
| **运单编号** | N/A | 全局唯一：waybill 在 `order.ltlWaybills` 数组中的位置 + 1 |
| **车牌格式** | 京A·UV001 | 京A·L001（下单时自动生成） |
| **compartments 缺失时** | N/A | `ensureLTLCompartments()` 从 waybills 动态重建，不依赖存储完整性 |

### LTL 交接记录显示规则（订单级多运单）

| 元素 | 多运单订单 | 单运单订单 |
|------|-----------|-----------|
| 交接记录标题 | `交接记录·N张运单` | `交接记录` |
| 运单行标签 | `运单X X舱` | `X舱` |
| 首页 Tab | `车牌 · N张` | `车牌` |
| 追踪 Tab | `车牌 · N张运单` | `车牌` |

## 跨页面一致性

- 所有业务线 Tab 的视觉样式统一（首页、运营看板使用相同的 Tab 组件或样式）
- FTL/LTL 的 L2 胶囊 Tab 样式完全对齐
- 所有"在途"类实时指标共享同一 store 数据源（`getActiveOrdersByType`）
- 首页动态面板的 Tab 选择与运营看板的 Tab 选择互不影响（独立状态）

## 下单页架构

> 下单流程步骤顺序，属于页面操作流。业务流程（状态机/按钮/通知）见 `business-flows.md`

- **整车配送页**：选车→编排经停点（发/收，支持 1→N 和 N→1）→货物配置→备注→费用
- **散件直送页**：添加运单列表（每张 = 投件地址 + 取件地址 + 货物）→系统自动拼车→费用（固定报价 + 拼车返利提示）
- 巡游页顺序：业务切换→车型→配套与数量(内联批量)→预期/巡检→期望时间→联系人→路线(可稍后)→备注→费用
- 批量：内联"+ 添加"模式，CSV 导入导出保留
- 车型：支持多选叠加，卡片右上角蓝色 ×N 徽章
